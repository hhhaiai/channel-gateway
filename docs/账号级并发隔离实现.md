# 账号级并发隔离实现

> 状态：实施文档
>
> 范围：限制单个 `channel/accountId` 同时执行的 Provider 请求数量。

## 1. 解决的问题

全局并发池只能限制整个服务。如果 Telegram 某个 bot 的待发送任务很多，它仍可能占满全部 lane，导致 QQ、Slack 或其他账号长期等待。

账号级隔离增加第二层边界：

```text
global max concurrency
  └─ per channel/account max concurrency
      └─ per destination conversation concurrency = 1
```

## 2. 账号键

```text
channel + accountId
```

相同平台的不同 bot 是不同账号；不同平台即使 accountId 都是 `default`，也不会共用计数。

## 3. 调度行为

假设：

```text
global = 8
perAccount = 2
```

Telegram bot-A 最多同时发送 2 条，Slack bot-B 和 QQ bot-C 仍可使用其他 lane。同一目标群始终只有 1 条 active send。

## 4. 配置

```json
{
  "deliveryMaxConcurrencyPerAccount": 2
}
```

约束：

```text
minimum = 1
maximum = 64
default = 2
```

全局并发小于账号上限时，全局值自然成为实际限制。

## 5. SQLite Claim

Worker 将已达到账号上限的账号传给 outbox：

```js
excludedAccounts: [{ channel: "telegram", accountId: "bot-a" }]
```

SQLite 只 claim 其他账号的 due delivery。这样不会先 lease 一条当前无法发送的任务，也不会让 lease 在内存队列中等待过期。

已被账号 exclusion 覆盖的 destination 不再重复生成 SQL destination exclusion，确保最大全局池下参数数量仍然有界。

## 6. 与后续速率限制的区别

本块限制同时进行的请求数量，不限制每秒请求数。后续独立实现：

- token bucket。
- Provider/account 每秒速率和 burst。
- 429 `Retry-After` cooldown。
- account/channel 异常状态和看板。

并发隔离是这些功能的前置基础，但不冒充完整限流器。
