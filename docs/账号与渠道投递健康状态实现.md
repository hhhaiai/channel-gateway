# 账号与渠道投递健康状态实现

## 1. 目的

投递系统原先只有全局数据库健康状态，无法回答“哪个平台、哪个发送账号异常”。本实现新增独立的 provider delivery 健康投影，让一个账号故障时可以被识别，同时不把整个 Channel Gateway 判定为不可用。

本块只提供状态数据和运行时接口。后台页面展示在下一独立提交完成。

## 2. 标识与隔离范围

账号唯一键为：

```text
channel + accountId
```

例如 `telegram/default` 与 `feishu/default` 是两个完全独立的状态。一个账号失败不会修改其他账号状态，也不会暂停其他账号的 worker lane。

渠道状态由该渠道下账号状态聚合，不额外建立一套可漂移的状态源。

## 3. 状态转换

```text
初始                       -> healthy
可重试失败                 -> degraded
终态失败                   -> unavailable
异常后第一次成功           -> recovering
recovering 后再次成功      -> healthy
任意新失败                 -> 清零恢复成功次数
```

已经处于 `unavailable` 的账号遇到后续可重试失败时仍保持 `unavailable`，避免严重程度被一次普通超时错误错误降低。

渠道聚合严重程度：

```text
unavailable > degraded > recovering > healthy
```

## 4. Durable 结果之后才更新

`DeliveryWorker` 的更新顺序是：

### 成功

1. provider send 成功。
2. `EventStore.completeDelivery()` 使用 lease token 完成 CAS。
3. 只有数据库确实返回已完成记录，才调用 `recordSuccess()`。

### 失败

1. provider send 抛出受控异常。
2. 计算 retry/terminal 状态。
3. `EventStore.retryDelivery()` 使用 lease token 完成 CAS。
4. 只有数据库确实写入 retry/failed 结果，才调用 `recordFailure()`。

因此 lease 过期、重复 worker 或 CAS 失败不会制造虚假的健康事件。

## 5. SQLite 积压统计

`EventStore.deliveryAccountStats()` 按目标账号聚合：

- `pending`
- `sending`
- `failed`
- 最早 `nextRetryAtMs`

查询只扫描 `pending/sending/failed`，不把大量已发送历史计入看板账号集合。SQLite 仍是当前 durable backlog 的事实源。

## 6. 输出字段

账号投影包含：

```json
{
  "channel": "telegram",
  "accountId": "bot-a",
  "status": "degraded",
  "errorCode": "RATE_LIMITED",
  "firstFailureAtMs": 1000,
  "lastFailureAtMs": 1500,
  "lastSuccessAtMs": null,
  "nextRetryAtMs": 5000,
  "pending": 4,
  "sending": 1,
  "failed": 0
}
```

不会输出：

- 消息正文
- 群名称或群成员
- provider 原始异常文本
- token、credential 或 request body

只接受受控大写错误码，防止异常内容被带入看板或未来通知系统。

## 7. 运行时边界

`createBridgeRuntime()` 暴露：

```js
runtime.deliveryHealth.snapshot()
```

瞬时状态转换当前保存在内存中；进程重启后异常时间线重新积累。SQLite 中的 backlog 和 failed 数量仍会恢复并出现在投影中，但不会把历史 failed 行自动解释为当前账号必然不可用，避免账号已恢复后被旧失败记录永久锁死。

如果未来需要跨实例共享异常时间线，应在存储接口抽象块中增加专用 health event/state repository，而不是把 Redis 逻辑直接写入 worker。

## 8. 测试覆盖

- 四态完整转换。
- unavailable 不会被普通可重试失败降级。
- 两个账号聚合为渠道最严重状态。
- SQLite 按账号统计积压和最早重试时间。
- Worker 只投影 durable CAS 成功的结果。
- snapshot 不包含 provider 原始异常内容。
