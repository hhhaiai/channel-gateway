# 投递健康后台看板实现

## 1. 目标

让管理员在现有 Channel Gateway 网页中直接看到：

- 哪个渠道异常。
- 哪个发送账号异常。
- 当前等待、发送中和终态失败数量。
- 下一次 durable retry 时间。
- 每账号 Token Bucket 的速率、剩余 token 和 burst。
- provider `Retry-After` 形成的 cooldown 截止时间。

该页面是只读运维视图，不提供手工覆盖健康状态、清除失败或绕过限流的入口。

## 2. API

### 请求

```http
GET /api/v1/delivery/status
Authorization: Bearer <gateway-token>
```

### 响应结构

```json
{
  "ok": true,
  "result": {
    "channels": [],
    "accounts": [],
    "rateLimits": []
  }
}
```

API 与其他 `/api/v1/*` 路由使用相同 Gateway Token 鉴权，并返回：

```http
Cache-Control: no-store
```

## 3. 脱敏边界

API 不直接透传 projection 或 limiter 对象。`api-handler.js` 对每一类记录显式挑选允许字段，因此即使内部对象以后增加以下内容，也不会自动暴露：

- provider 原始错误文本
- 消息正文
- request body
- token、credential、authorization
- 群成员或用户身份

账号错误只显示类似 `RATE_LIMITED`、`TIMEOUT` 的受控错误码。

## 4. 页面加载逻辑

点击“连接并刷新”后并行请求：

```text
/api/v1/links/config
/api/v1/channels?probe=true
/api/v1/delivery/status
```

三者全部成功后统一刷新页面，避免一个旧状态与一个新配置被拼成同一画面。Token 仍只保存在当前页面 JavaScript 内存，不写 `localStorage` 或 Cookie。

## 5. 渠道视图

每个有投递数据的渠道显示：

- 渠道 ID
- 中文状态文字
- 账号数量
- `pending/sending/failed` 数量

状态同时有文字和左边框样式，不仅依赖颜色：

```text
healthy      正常
recovering   恢复中
degraded     异常（可重试）
unavailable  不可用
```

## 6. 账号视图

账号表格按 projection 已排序的数据展示：

- `channel / accountId`
- 健康状态
- 最后受控错误码
- durable backlog
- 下一次重试时间
- 每秒速率、当前 token、burst
- cooldown 截止时间

尚未产生发送活动的账号没有 Token Bucket runtime state，此时明确显示“尚无活动采样”，而不是伪造 token 数值。

## 7. 安全实现

- 页面全部使用 `textContent` 和 DOM API。
- 不使用 `innerHTML` 渲染运行时数据。
- 继续使用严格 CSP：`default-src 'none'`。
- CSS/JS/HTML 均为本地静态资源，无第三方前端依赖。
- 页面没有健康状态修改接口。

## 8. 资源影响

- 页面刷新才执行 snapshot，不增加定时轮询。
- SQLite 账号聚合由前一块提供，只统计非 sent delivery。
- limiter snapshot 只遍历已活动账号。
- API 不复制消息正文或大型 metadata。

## 9. 测试覆盖

- API 返回渠道、账号和 limiter 状态。
- API 去除内部对象额外的 secret 字段。
- 响应禁止缓存。
- HTML 存在投递健康区域和账号表格。
- JavaScript 加载 `/delivery/status` 并调用 `renderDeliveryHealth()`。
- 原有控制台、API、runtime 和完整测试保持通过。

## 10. 后续

通知系统未来可消费同一个脱敏状态 contract，但应独立实现状态变化事件、负责人映射与通知抑制，不能在当前页面刷新路径中直接发送通知。
