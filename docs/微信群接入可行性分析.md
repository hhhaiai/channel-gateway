# Channel Gateway 微信群接入可行性分析

> 核查日期：2026-07-13。结论基于当前仓库、固定 `openclaw@2026.6.11`、腾讯/OpenClaw 在线文档、npm 最新包和实际包源码检查。

## 1. 结论

必须先区分三种“微信群”：

| 目标 | 当前结论 | 是否建议正式接入 |
| --- | --- | --- |
| 普通个人微信群 | 技术上可用桌面 RPA/hook 接入，但没有可承诺生产稳定的官方 provider | **不建议生产**；只允许隔离测试号 PoC |
| 企业微信成员群聊 | 企业微信智能机器人支持群聊入站与受控主动推送，和当前架构匹配 | **推荐正式接入** |
| 企业微信客户群（群内包含普通微信客户） | 官方提供群资料、变更事件、人工确认群发和合规会话存档，但没有完整实时双向 bot 能力 | 不应承诺为实时群机器人 |

因此：

- 如果业务必须是现有的普通个人微信群，当前只能做高风险 PoC，不能承诺长期稳定。
- 如果业务可以迁移到企业微信群，建议接腾讯企业微信团队维护的 `@wecom/wecom-openclaw-plugin`。
- 腾讯 Weixin 团队的 `@tencent-weixin/openclaw-weixin` 当前只能接微信私聊，不能通过配置开启微信群。

## 2. 腾讯官方 Weixin 插件为什么不能接群

当前在线最新包：

```text
@tencent-weixin/openclaw-weixin@2.4.6
channel id: openclaw-weixin
OpenClaw requirement: >=2026.5.12
```

本仓库固定的 `openclaw@2026.6.11` 满足版本要求，外置路径也能通过当前 launcher 检查。但插件能力明确是：

```ts
capabilities: {
  chatTypes: ["direct"],
  media: true,
}
```

实现中同时存在这些硬边界：

- 入站 `ChatType` 固定为 `direct`；
- 权限检查固定 `isGroup: false`；
- 路由固定 `peer.kind = direct`；
- outbound target 只识别 `xxx@im.wechat` 用户 ID；
- 协议类型虽然保留 `group_id` 字段，但当前业务逻辑没有使用；
- OpenClaw 在线文档明确写明 capability metadata 只声明 direct chat，未声明 group chat。

底层腾讯 iLink API 当前插件实现的是：

```text
getUpdates  → 长轮询私聊消息
sendMessage → 向用户 ID 发送文本/图片/视频/文件
```

不能只把 `chatTypes` 改成 `group`。还缺少服务端群事件/群发送能力证明，以及群 ID、成员 ID、@状态、群准入、上下文 token 和群 receipt 的完整实现。

## 3. 企业微信群为什么适合当前架构

腾讯企业微信智能机器人长连接提供：

- `wss://openws.work.weixin.qq.com`；
- 群里 `@机器人` 后产生带 `chatid`、`chattype=group`、`from.userid` 的回调；
- `aibot_send_msg` 可主动向已激活的用户或群聊推送消息；
- 不需要公网 webhook，适合当前独立服务；
- 同一机器人同时只有一个有效长连接，和当前“一个群一个 bot owner”的单 owner 模型一致。

官方限制：

- 群里通常只有 `@机器人` 的消息会进入机器人回调，不是监听所有群消息；
- 必须先由用户在对应会话给机器人发消息，之后才能主动推送到该会话；
- 同一会话的回复和主动推送合计限制为 30 条/分钟、1000 条/小时；
- 长连接需要 ping、断线重连、单连接租约和消息幂等。

### 3.1 和当前模型的字段映射

```text
channel        = wecom
accountId      = OpenClaw WeCom 账号键（单账号默认 default，不是 Bot ID）
conversationId = 企业微信 chatid（保留原始大小写）
to             = 同一真实 chatid
messageId      = 入站 canonical messageId 使用企业微信 msgid
sender.id      = from.userid
isGroup        = true
```

主动发送的 receipt/message id 可能来自插件返回的 `req_id`，必须和入站 `msgid` 分开记录。

当前实现已经具备：

- provider-neutral canonical event；
- `channel + accountId + conversationId` 精确匹配；
- 一个群只能由一个 bot account 拥有；
- durable outbox、重试、账号健康状态、SSE/ACK；
- 同 destination 串行和 account 级并发/速率限制；
- 群媒体、reply/receipt 的扩展字段边界。

所以不需要改 `route-links.js` 或 EventStore 主模型。

### 3.2 已完成的 launcher 兼容

腾讯企业微信团队当前 npm 包为：

```text
@wecom/wecom-openclaw-plugin
plugin manifest id: wecom-openclaw-plugin
channel id: wecom
capabilities.chatTypes: ["direct", "group"]
```

旧 launcher 曾错误假定 `manifest.id === channel id`，表现为：

```text
Error: @wecom/wecom-openclaw-plugin manifest must declare channel id wecom-openclaw-plugin
```

当前实现已经完成以下兼容改造：

1. 外置插件 descriptor 分开保存 `pluginId` 与 `channelIds`；
2. `plugins.entries` 使用 `pluginId` 启用插件；
3. 路由 endpoint 使用真实 channel id `wecom`；
4. manifest 可声明一个或多个 channel，不再强制 plugin id 等于 channel id；
5. 增加 `pluginId != channelId`、重复 plugin id 和重复 Channel ownership 的 launcher 回归测试；
6. 控制台 catalog 增加 `wecom`，并把腾讯微信私聊显示为 `openclaw-weixin`，避免两者混淆；
7. `package.json` 精确锁定 `@wecom/wecom-openclaw-plugin@2026.5.25`，`all` profile 可自动发现，不跟随异常的 `latest` 标签漂移；
8. rich hook patch v2 会把插件提供的 `ChatType: "group" | "channel"` 映射为 canonical `isGroup: true`，并能在核验 v1 marker 的文件 hash 与精确 postcondition 后升级。

### 3.3 推荐初始限流

当前 limiter 是 account 级，不是 conversation 级。为避免触发企业微信每会话 1000 条/小时限制，首期可对 WeCom 账号使用保守值：

```json5
{
  deliveryAccountRateLimits: [
    { channel: "wecom", accountId: "default", ratePerSecond: 0.25, burst: 1 }
  ]
}
```

这会把整个账号限制为约 15 条/分钟、900 条/小时，安全但对多群较保守。规模扩大前应增加 destination/conversation 级 token bucket，而不是盲目提高账号总速率。

## 4. 普通个人微信群的现实方案

### 4.1 `@agent-wechat/wechat`：最短 PoC 路径

当前包：

```text
@agent-wechat/wechat@0.11.15
channel id: wechat
chatTypes: ["direct", "group"]
group id: xxx@chatroom
```

它已经是 OpenClaw Channel plugin，且 manifest id 与 channel id 都是 `wechat`，实测可以通过当前 `CHANNEL_GATEWAY_PLUGIN_PATHS` discovery。已实现群策略、`requireMention`、群成员 allowlist、群目录和群消息收发。

但底层不是腾讯官方机器人 API，而是：

```text
Channel Gateway/OpenClaw
  → agent-wechat REST service
  → Linux WeChat Desktop + Xvfb
  → AT-SPI/UI automation/ptrace
```

运行容器需要：

```yaml
security_opt:
  - seccomp=unconfined
cap_add:
  - SYS_PTRACE
```

主要风险：

- 需要完整桌面微信会话和手机登录状态；
- 不能运行在 Fargate、Cloud Run 等受限/serverless 环境；
- 微信客户端 UI 更新可能立即破坏自动化；
- npm/GitHub 当前没有明确 license，不能直接用于商业交付；
- 属于非腾讯授权自动化，存在账号风控和协议风险；
- 不应使用主微信号或重要群做首次验证。

因此只能作为隔离 VM、测试微信号、文本 + 群 @ 的 PoC。

### 4.2 WCFerry：Windows hook 备选

WCFerry 能提供 `roomid`、sender、message id、文本/媒体和群成员能力，字段很容易映射到当前 DTO。但仓库当前已 archived，属于 Windows WeChat hook，并且版本与微信客户端强绑定。它适合研究/PoC，不适合生产承诺。

如果做 sidecar，映射建议：

```text
channel        = wechat-personal
accountId      = bot wxid
conversationId = roomid（xxx@chatroom）
to             = roomid
messageId      = WxMsg.id
sender.id      = WxMsg.sender
```

需要额外处理：

- 丢弃 `from_self()`；
- provider message id 去重，不能只依赖不可见 marker；
- outbound 只有状态码时只能记为 `accepted`，不能记为可靠 `delivered`；
- 账号并发从 1 开始；
- 微信版本、登录失效、GUI/注入异常必须进入熔断和健康状态。

## 5. 不同方案的最终选择

| 需求 | 推荐方案 |
| --- | --- |
| 微信私聊 | 腾讯 `@tencent-weixin/openclaw-weixin` |
| 企业微信成员群 | 腾讯企业微信官方 `@wecom/wecom-openclaw-plugin@2026.5.25`；当前 launcher 已兼容并自动发现 |
| 企业微信客户群实时机器人 | 当前不承诺；先做真实租户 capability probe |
| 普通个人微信群 PoC | 隔离测试号 + `@agent-wechat/wechat`，或 Windows WCFerry sidecar |
| 普通个人微信群生产 | 当前不建议立项为“稳定能力” |

## 6. 企业微信群正式接入验收顺序

1. 使用 `all` profile 安装已锁定的 `@wecom/wecom-openclaw-plugin@2026.5.25`，确认 launcher 生成 `plugins.entries["wecom-openclaw-plugin"]`。
2. 停止 Gateway，使用同一隔离数据目录配置 Bot ID/Secret，并确认 links 使用 `channel: "wecom"`。
3. 重新启动 Gateway 后运行 `channels status --probe`，确认 `gatewayReachable: true` 且账号状态可用。
4. 建立 Bot WebSocket，验证单连接、ping、断线重连和状态 probe。
5. 在测试企业微信群中 `@机器人`，确认 canonical event 的 `conversationId` 等于原始 `chatid`。
6. 使用 `/api/v1/messages` 对同一 chatid 做受控发送。
7. 写入一个 WeCom + Telegram/Feishu 的双 endpoint link 并重启。
8. 双向验证、echo suppression、receipt、媒体和 reply。
9. 配置保守速率，执行 429/cooldown、断线和重启恢复验证。
10. WeCom 插件默认 `groupPolicy` 是 `open`。取得测试群 `chatid` 后必须显式设置
    `channels.wecom.groupPolicy: "allowlist"` 和 `groupAllowFrom: ["<CHAT_ID>"]`，重启并验证未批准群被拒绝；
    验证完成后再逐个扩大群 allowlist。
