# 存储接口抽象与大数据迁移边界

## 1. 目的

Channel Gateway 当前继续使用 SQLite，但 Runtime、Worker 和 API 不再以 SQLite class 身份作为依赖。它们依赖版本化 JavaScript storage contract，未来可以替换为 Redis、大数据存储或组合 adapter。

默认行为没有变化：未传入 `store` 时仍创建 `EventStore(databasePath)`。

## 2. Contract 版本

```js
GATEWAY_STORAGE_CONTRACT_VERSION === 1
```

SQLite 声明：

```js
store.storageCapabilities()
// {
//   contractVersion: 1,
//   backend: "sqlite",
//   durable: true,
//   atomicFanout: true,
//   aggregateLeases: true,
//   transformLeaseCas: true
// }
```

如果 adapter 声明了 capability，其 contract version 必须匹配。未声明 capability 的旧测试/轻量 adapter 仍可按方法集合验证，便于渐进迁移。

## 3. 能力组

### Core ingress

所有 adapter 必须实现：

```text
enqueue
findEcho
findDeliveryByMarker
resolveReplyTargets
close
```

### Delivery worker

存在自动 links 和 sender 时额外要求：

```text
claimNextDelivery
completeDelivery
retryDelivery
saveDeliveryTransform
```

### HTTP API

启用 Gateway API 时额外要求：

```text
listPending
ack
fail
pendingCount
deliveryCounts
```

辅助能力 `deliveryAccountStats`、`prune`、`on/off` 当前为可选；Runtime 已有安全 fallback。实现生产 adapter 时仍建议全部提供。

缺少方法会在 Runtime 构建阶段一次性列出，而不是运行到某条消息时才报 `is not a function`：

```text
code: INVALID_STORAGE_ADAPTER
storage adapter is missing methods: ...
```

## 4. 方法签名不是完整合同

外部 adapter 必须同时满足以下行为语义。

### `enqueue(event, { deliveries })`

- Event 与全部 fan-out delivery 原子提交。
- 相同 event id 幂等，不重复 fan-out。
- enrich 已有 event 时不能清除非空字段。
- 提交成功后才能发布 pending notification。

### `claimNextDelivery(options)`

- 对 due delivery 执行 CAS lease。
- 同一 delivery 不能被两个 worker 同时领取。
- 支持 destination/account exclusion。
- 已形成 aggregate 必须恢复相同成员及顺序。
- 返回 request、lease token、attempt 和 aggregate member ids。

### `completeDelivery`

- 必须验证当前 lease token。
- Aggregate 全部成员原子完成。
- Provider receipt 只关联 leader，或提供等价的唯一关系模型。

### `retryDelivery`

- 必须验证当前 lease token。
- Aggregate 全成员共享重试状态、错误码和 due time。
- Max attempts 终态转换原子完成。

### `saveDeliveryTransform`

- 只允许当前 leader lease 写入。
- 相同内容可幂等重复写入。
- 不允许覆盖已经持久化的不同 Transform 输出。
- Retry/restart 必须返回相同 `transformedRequest`。

## 5. Runtime 注入

```js
const runtime = createBridgeRuntime({
  store: customStorageAdapter,
  links,
  sender,
});
```

测试已使用一个委托 wrapper 包装 SQLite，证明 Runtime 不读取 `instanceof EventStore`，也不依赖 SQLite 私有字段。

## 6. Redis / 大数据建议分层

不建议把所有方法简单映射到一个 Redis Hash。推荐组合 adapter：

```text
Durable event/archive      -> 大数据或关系型存储
Due delivery index         -> Redis sorted set / queue
Lease/CAS                  -> Redis Lua 或具备事务的队列系统
Idempotency/echo index     -> Redis + durable archive
Analytics                  -> 异步写入大数据系统
GatewayStorageAdapter      -> 对业务层实现统一合同
```

关键原则：大数据分析写入不能处于 provider send 的同步关键路径；Redis 加速层也不能成为唯一审计副本，除非部署明确接受其持久化和灾备模型。

## 7. 一致性要求

未来分布式实现至少需要：

- Event id 全局唯一。
- Delivery id 全局唯一。
- Lease token fencing。
- 服务端时间或可比较的统一时钟。
- Aggregate member 顺序稳定。
- Transform request write-once。
- Receipt 唯一索引。
- ACK/prune 不得删除仍被 delivery 引用的 event。

只实现相同方法名但不满足这些条件，不视为兼容 adapter。

## 8. 迁移步骤

1. 实现 adapter 和 `storageCapabilities()`。
2. 复用 `EventStore` contract tests 建立兼容测试套件。
3. 双写阶段只用于比对，不让两个后端同时驱动 provider send。
4. 校验 event/delivery/aggregate/receipt 数量与哈希。
5. 停止旧 worker，等待 active lease 结束或显式 fencing。
6. 切换单一 active storage adapter。
7. 保留 SQLite 只读回滚窗口。

“不要主备模式”的业务要求仍适用：迁移可以双写校验，但不能让两个存储后端同时作为 active delivery owner。

## 9. 测试覆盖

- SQLite capability/version 声明。
- 全功能合同验证。
- 缺少方法的完整错误列表。
- Event-only adapter 不被迫实现 Worker 方法。
- 委托型非 SQLite identity adapter 可运行 Runtime ingress。
- 原有 SQLite、聚合、Transform、API 和 Worker 测试保持通过。
