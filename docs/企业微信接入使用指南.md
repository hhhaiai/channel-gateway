# Channel Gateway 企业微信接入使用指南

> 适用版本：`channel-gateway` 当前 `main`、`openclaw@2026.6.11`、`@wecom/wecom-openclaw-plugin@2026.5.25`。

## 1. 可用性结论

当前实现已经完成企业微信官方插件的加载、配置、群事件识别和 `links` 路由适配：

- OpenClaw plugin ID：`wecom-openclaw-plugin`；
- Channel Gateway Channel ID：`wecom`；
- 支持企业微信成员群的群内 `@机器人` 入站；
- 支持向已激活的企业微信会话主动发送；
- 可与 Telegram、飞书、Slack、QQ 等其它已加载 Channel 组成双向互通房间；
- rich hook patch v2 会把 WeCom 的 `ChatType: "group"` 映射为 canonical `isGroup: true`；
- SQLite durable outbox、重试、账号限流、断线后的 pending 恢复和防回环逻辑继续复用现有实现。

仓库自动测试和插件加载验证只能证明软件基线稳定。正式上线前仍必须使用真实企业微信租户完成本文第 9 节验收。

## 2. 前置条件

准备以下内容：

1. Node.js `22.22.2`。低于该版本不属于正式运行环境。
2. 企业微信智能机器人的 Bot ID 和 Bot Secret。
3. 一个隔离的测试企业微信群。
4. 至少一个准备互通的其它 Channel，例如 Telegram 或飞书。
5. 独立的 `CHANNEL_GATEWAY_DATA_DIR` 和强随机 `CHANNEL_GATEWAY_TOKEN`。

Bot Secret 只在本机交互式 onboarding 中输入，不写进 Git、links、命令行参数、浏览器存储或工单。

## 3. 安装固定版本

在仓库根目录执行：

```bash
cd /Users/sanbo/Desktop/openclaw
node --version

npm ci --omit=dev --include=optional
npm run verify:openclaw-patch
npm ls @wecom/wecom-openclaw-plugin @wecom/aibot-node-sdk
```

应看到：

```text
@wecom/wecom-openclaw-plugin@2026.5.25
@wecom/aibot-node-sdk@1.0.7
```

`base` profile 不安装 WeCom；本地 `--include=optional` 和 Docker `all/common` profile 才安装。

## 4. 准备隔离实例

```bash
export CHANNEL_GATEWAY_DATA_DIR="$PWD/.channel-gateway"
export CHANNEL_GATEWAY_BIND=loopback
export CHANNEL_GATEWAY_PORT=18789

# 生产环境改由 secret manager 注入。
export CHANNEL_GATEWAY_TOKEN="$(openssl rand -hex 32)"
```

首次启动一次，让 launcher 生成目录和初始配置：

```bash
npm start
```

确认 `/healthz` 可访问后按 `Ctrl-C` 停止 Gateway，再导出 OpenClaw 原生命令使用的同一实例环境：

```bash
export OPENCLAW_HOME="$CHANNEL_GATEWAY_DATA_DIR"
export OPENCLAW_CONFIG_PATH="$CHANNEL_GATEWAY_DATA_DIR/config/openclaw.json"
export OPENCLAW_STATE_DIR="$CHANNEL_GATEWAY_DATA_DIR/state"
export OPENCLAW_WORKSPACE_DIR="$CHANNEL_GATEWAY_DATA_DIR/workspace"
export OPENCLAW_OAUTH_DIR="$CHANNEL_GATEWAY_DATA_DIR/credentials"
export OPENCLAW_GATEWAY_TOKEN="$CHANNEL_GATEWAY_TOKEN"
export OPENCLAW_GATEWAY_PORT="$CHANNEL_GATEWAY_PORT"
```

新实例应自动包含：

```json5
{
  plugins: {
    entries: {
      "wecom-openclaw-plugin": { enabled: true }
    }
  }
}
```

如果是已有 `base/all/common` 实例，launcher 会保留旧配置并输出 WeCom 未启用告警，不会阻断原 Channel 启动。请先按照 [`操作手册.md`](../操作手册.md) 6.1 节完成备份和幂等迁移。

## 5. 配置 Bot ID 和 Secret

保持 Gateway 停止，执行：

```bash
./node_modules/.bin/openclaw channels add
```

在交互界面中选择 **WeCom**，然后输入 Bot ID 和 Bot Secret。完成后启动 Gateway：

```bash
npm start
```

另开终端，重新导出第 4 节环境变量，再执行：

```bash
./node_modules/.bin/openclaw plugins doctor
./node_modules/.bin/openclaw channels status --probe
```

只有 Gateway 运行时的 probe 才能证明账号连通；Gateway 停止时出现的 `configOnly` 不能作为验收结果。

## 6. 获取测试群 chatid 并关闭默认开放策略

WeCom 插件默认 `groupPolicy` 是 `open`。如果平台后台能够提前取得测试群 `chatid`，应在启动前直接配置 allowlist。

如果不能提前取得，只允许在隔离测试租户中执行一次发现流程：

1. 在测试企业微信群中 `@机器人` 发送一条唯一文本。
2. 读取 canonical event：

```bash
curl -fsS \
  -H "Authorization: Bearer $CHANNEL_GATEWAY_TOKEN" \
  "http://127.0.0.1:${CHANNEL_GATEWAY_PORT}/api/v1/events?limit=100"
```

3. 从对应事件记录以下字段：

```text
channel        = wecom
accountId      = default（或实际 OpenClaw account key）
conversationId = 企业微信原始 chatid
messageId      = 入站 msgid
```

4. 停止 Gateway，显式收紧群策略：

```bash
export WECOM_CHAT_ID='<真实 chatid>'

./node_modules/.bin/openclaw config set \
  channels.wecom.groupPolicy '"allowlist"' --strict-json

./node_modules/.bin/openclaw config set \
  channels.wecom.groupAllowFrom "[\"$WECOM_CHAT_ID\"]" --strict-json

./node_modules/.bin/openclaw config validate
```

5. 重新启动 Gateway，确认批准群仍可触发，未批准群被拒绝。

`groupAllowFrom` 保存的是群 `chatid`，不是成员 userid。只填写 `groupAllowFrom` 而不设置 `groupPolicy: "allowlist"` 不会形成群限制。

## 7. 验证主动发送

企业微信会话必须先联系机器人，平台才可能允许主动推送。使用同一个真实 `chatid` 发送受控测试消息：

```bash
export WECOM_CHAT_ID='<真实 chatid>'

curl -fsS \
  -H "Authorization: Bearer $CHANNEL_GATEWAY_TOKEN" \
  -H 'Content-Type: application/json' \
  --data "$(node -e '
    process.stdout.write(JSON.stringify({
      channel: "wecom",
      accountId: "default",
      to: process.env.WECOM_CHAT_ID,
      text: "Channel Gateway 企业微信发送测试",
      idempotencyKey: `wecom-smoke-${Date.now()}`
    }))
  ')" \
  "http://127.0.0.1:${CHANNEL_GATEWAY_PORT}/api/v1/messages"
```

同时确认企业微信群实际收到消息。HTTP 成功但群内未收到不能判定通过。

## 8. 配置与其它软件双向互通

推荐通过控制台编辑：

```text
http://127.0.0.1:18789/channel-gateway
```

输入 Gateway Bearer token，创建一个互通房间并加入 WeCom 与其它 Channel。WeCom + Telegram 示例：

```json5
{
  id: "wecom-telegram",
  endpoints: [
    {
      id: "wecom-group",
      channel: "wecom",
      accountId: "default",
      conversationId: "<WECOM_CHAT_ID>",
      to: "<WECOM_CHAT_ID>",
      receive: true,
      send: true
    },
    {
      id: "telegram-group",
      channel: "telegram",
      accountId: "default",
      conversationId: "-1001234567890",
      to: "-1001234567890",
      receive: true,
      send: true
    }
  ]
}
```

保存 links 后必须重启 Gateway。保存只原子更新配置，不会热加载路由拓扑。

建议为首个 WeCom 账号设置保守限流：

```json5
{
  deliveryAccountRateLimits: [
    {
      channel: "wecom",
      accountId: "default",
      ratePerSecond: 0.25,
      burst: 1
    }
  ]
}
```

这会限制整个 WeCom account，而不是单个群。群数量扩大前应根据真实平台限流结果评估 conversation 级 limiter。

## 9. 上线前验收

逐项完成并记录证据：

- [ ] Node.js 精确使用 `22.22.2`。
- [ ] rich hook patch verification 通过。
- [ ] `plugins doctor` 无 WeCom 加载错误。
- [ ] `channels status --probe` 在 Gateway 运行时显示账号可用。
- [ ] 测试群 `@机器人` 后产生 `channel=wecom`、正确 `chatid`、`isGroup=true` 的事件。
- [ ] `groupPolicy=allowlist`，只允许批准的 `chatid`。
- [ ] `/api/v1/messages` 成功且群内实际收到消息。
- [ ] WeCom → 其它 Channel 转发成功。
- [ ] 其它 Channel → WeCom 转发成功。
- [ ] 同一消息不会无限回环或重复放大。
- [ ] Gateway 重启后 pending delivery 能继续恢复。
- [ ] 断网重连、429/cooldown 和平台限流行为已经验证。
- [ ] 文本、媒体、reply/receipt 按业务实际需要分别验收。
- [ ] Bot Secret、Gateway token、SQLite 数据和群消息日志符合企业的数据权限要求。

## 10. 主要边界

- 本指南接入的是企业微信成员群，不是普通个人微信群。
- 企业微信客户群不应直接承诺具备完整实时双向 bot 能力。
- 群入站通常要求 `@机器人`，不能把它视为监听所有群普通消息。
- `links` 是消息路由配置，不是多租户授权边界；不可信租户必须拆分实例。
- 仓库测试不能替代真实租户、网络、凭据、群权限、媒体和限流验收。

更完整的统一 API、SSE/ACK、links revision 更新和逐 Channel 操作见 [`逐渠道对接操作指南.md`](../逐渠道对接操作指南.md)。
