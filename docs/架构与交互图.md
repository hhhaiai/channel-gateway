# Channel Gateway：拓扑、交互、框架与调用逻辑图

> 图中一个 **link（互通房间）** 可包含 N 个 endpoint。endpoint 是一个实际 QQ/TG 群（`channel + accountId + conversationId + to`）。因此无需为 QQ↔TG 单独写规则：任一 endpoint 入站后 fan-out 到同 link 的其他可发送 endpoint。

## 1. 拓扑图：1↔N 与 N↔N

```mermaid
flowchart LR
  subgraph L[互通房间：team-main]
    Q1[QQ群 A\nqqbot / qq-a]
    Q2[QQ群 B\nqqbot / qq-b]
    QN[QQ群 N\nqqbot / qq-n]
    T1[TG 群 1\ntelegram / -1001]
    T2[TG 群 2\ntelegram / -1002]
    TN[TG 群 N\ntelegram / -100n]
  end
  Q1 <--> G[Channel Gateway\nlink fan-out]
  Q2 <--> G
  QN <--> G
  T1 <--> G
  T2 <--> G
  TN <--> G
```

**规则：** 任一个群发消息，Gateway 向该房间其它 `send:true` endpoint 投递；发送群自身不会收到回环。删去任意 endpoint 即不再参与该房间。

## 2. 交互图：一条消息如何转发

```mermaid
sequenceDiagram
  participant A as 发言群（QQ 或 TG）
  participant C as OpenClaw Channel
  participant B as Bridge Runtime
  participant S as SQLite Event/Outbox
  participant W as Delivery Worker
  participant D as 其余 QQ/TG 群

  A->>C: 入站消息
  C->>B: message_received（关联信息）
  C->>B: before_dispatch（完整消息）
  B->>B: normalize + matchSourceEndpoints + planFanout
  B->>S: 同一事务写 event + N-1 deliveries
  B-->>C: handled（不进入 Agent/LLM）
  W->>S: lease pending delivery
  loop 每个目的 endpoint
    W->>C: Gateway RPC send
    C->>D: Provider API 发送
    C-->>W: receipt / messageId
    W->>S: sent 或 retry/failed
  end
```

## 3. 框架图：运行组件和信任边界

```mermaid
flowchart TB
  Browser[浏览器控制台\n/channel-gateway\nToken 仅内存] -->|Bearer| Api[/api/v1/*\nGateway auth]
  Browser -. 静态 HTML/CSS/JS .-> Static[plugin-managed 静态资产]
  Api --> ConfigSvc[Links Config Service]
  ConfigSvc -->|mutateConfigFile base=source| HostCfg[OpenClaw Host Config Writer\nJSON5 / lock / backup / atomic write]
  Api --> Runtime[Bridge Runtime]
  Runtime --> SQLite[(SQLite\nevents + deliveries)]
  Runtime --> Rpc[Gateway RPC]
  Worker[Delivery Worker] --> SQLite
  Worker --> Rpc
  Rpc --> Channels[Telegram / QQBot / 其它 Channel]
  Channels --> Groups[各群 endpoint]
```

静态控制台不携带配置或凭据；可写 API 必须经过 Gateway Bearer token。控制台只写 `plugins.entries.channel-gateway.config.links`，不会写平台 token、AppSecret、session 或 OAuth/二维码凭据。

## 4. 代码调用逻辑图

```mermaid
flowchart TD
  Start[bin/channel-gateway.js] --> Launcher[src/launcher/*\n隔离 config/state/credentials]
  Launcher --> Host[openclaw Gateway]
  Host --> Plugin[src/plugin.js]
  Plugin --> Runtime[src/bridge-runtime.js]
  Plugin --> Console[src/console-assets.js + ui/*]
  Plugin --> Config[src/links-config-service.js]
  Runtime --> Ingress[onMessageReceived / onBeforeDispatch]
  Ingress --> Normalize[src/event-normalizer.js]
  Normalize --> Links[src/route-links.js\ncompileLinks / planFanout]
  Links --> Store[src/event-store.js]
  Store --> Worker[src/delivery-worker.js]
  Worker --> Sender[src/self-api-sender.js]
  Sender --> Rpc[src/gateway-rpc.js]
  Runtime --> Api[src/api-handler.js]
  Api --> Config
  Api --> Store
```

## 管理页保存流程

```mermaid
sequenceDiagram
  participant UI as 控制台
  participant API as api-handler
  participant CS as links-config-service
  participant Host as api.runtime.config

  UI->>API: GET /api/v1/links/config
  API->>CS: read()
  CS-->>UI: links + revision
  UI->>API: PUT /api/v1/links/config {links, revision}
  API->>CS: update()
  CS->>CS: compileLinks() + schema boundary
  CS->>Host: mutateConfigFile(base: source, afterWrite: none)
  Host-->>CS: atomic durable write
  CS-->>UI: restartRequired: true
  UI-->>UI: 提示重启 channel-gateway
```
