# Channel Gateway 逐渠道对接操作指南

> 目标：按“一个 Channel 一次”的方式完成真实账号对接。每个平台都必须依次通过：**插件已加载 → 账号可用 → 单渠道发送成功 → 真实入站事件可见 → 加入 links 后跨渠道转发成功**。不要一次配置多个平台后再集中排错。

当前仓库固定使用 `openclaw@2026.6.11`。Channel Gateway 复用 OpenClaw 的 Channel 插件完成登录、收消息和发消息；本项目只提供统一的 REST/SSE 控制面、事件持久化和跨渠道 durable fan-out，不替代各平台原生 onboarding。

## 1. 首次准备

### 1.1 安装 profile

在当前仓库执行：

```bash
cd /Users/sanbo/Desktop/openclaw

# 只准备 core Channel（例如 Telegram）
# npm ci --omit=dev --omit=optional

# 准备 package.json 中全部 optional Channel（含固定版本企业微信插件）
npm ci --omit=dev --include=optional

npm run verify:openclaw-patch
```

说明：

- `base` 不安装 optional Channel。
- `all`/`common` 会安装 `package.json` 当前声明的全部 optional Channel；是否真正可用仍取决于凭据、平台权限、sidecar、webhook 和运行时 probe。
- `CHANNEL_GATEWAY_PLUGIN_PATHS` 只用于自定义安装位置或本 profile 未包含的外置插件，必须填写绝对路径。多个路径在 macOS/Linux 用 `:` 分隔。
- 任何已有实例（包括旧版 `all/common`）都不会因升级后新安装 WeCom 而被 launcher 静默改写。若配置尚未启用该插件，launcher 会告警但继续启动，WeCom 保持不可用；按《操作手册》6.1 的备份与幂等迁移步骤补齐 load path/entry 后才会启用。

### 1.2 固定实例目录、Token 和端口

```bash
export CHANNEL_GATEWAY_DATA_DIR="$PWD/.channel-gateway"
export CHANNEL_GATEWAY_BIND=loopback
export CHANNEL_GATEWAY_PORT=18789

# 生产环境应由 secret manager 注入。首次本地对接可显式生成并妥善保存。
export CHANNEL_GATEWAY_TOKEN="$(openssl rand -hex 32)"
```

首次启动：

```bash
npm start
```

另开终端确认进程可达：

```bash
curl -fsS http://127.0.0.1:18789/healthz
```

launcher 会建立：

```text
$CHANNEL_GATEWAY_DATA_DIR/
├── config/openclaw.json
├── state/channel-gateway.sqlite
├── credentials/gateway-token
└── workspace/
```

如果启动时没有显式传入 `CHANNEL_GATEWAY_TOKEN`，launcher 会生成 `0600` 的 token 文件。之后只在受信任 shell 中重新加载，不要打印到日志：

```bash
export CHANNEL_GATEWAY_DATA_DIR="${CHANNEL_GATEWAY_DATA_DIR:-$PWD/.channel-gateway}"
export CHANNEL_GATEWAY_TOKEN="$(cat "$CHANNEL_GATEWAY_DATA_DIR/credentials/gateway-token")"
```

首次目录生成完成后按 `Ctrl-C` 停止 Gateway，再进行平台 onboarding。不要让 onboarding 命令和运行中的 Gateway 同时修改同一份配置或 credentials。

### 1.3 让 OpenClaw 原生命令使用同一个实例

每次执行 `openclaw channels add/login/status` 前都先导出：

```bash
export OPENCLAW_HOME="$CHANNEL_GATEWAY_DATA_DIR"
export OPENCLAW_CONFIG_PATH="$CHANNEL_GATEWAY_DATA_DIR/config/openclaw.json"
export OPENCLAW_STATE_DIR="$CHANNEL_GATEWAY_DATA_DIR/state"
export OPENCLAW_WORKSPACE_DIR="$CHANNEL_GATEWAY_DATA_DIR/workspace"
export OPENCLAW_OAUTH_DIR="$CHANNEL_GATEWAY_DATA_DIR/credentials"
export OPENCLAW_GATEWAY_TOKEN="$CHANNEL_GATEWAY_TOKEN"
export OPENCLAW_GATEWAY_PORT="${CHANNEL_GATEWAY_PORT:-18789}"
```

通用交互入口：

```bash
./node_modules/.bin/openclaw channels add
```

`channels add/login` 在 Gateway 停止时写配置；完成后先重新启动 Gateway，再执行
`channels status --probe`。Gateway 停止时该命令只能返回 `configOnly` 状态，不能证明账号真实可用。

## 2. 每个 Channel 都按这 10 步对接

### 第 1 步：确认安装来源

- core Channel：直接配置，例如 Telegram。
- optional Channel：确认相应目录存在，例如 `node_modules/@openclaw/feishu`。
- 外置 Channel：固定版本、审查来源和许可后，通过 `CHANNEL_GATEWAY_PLUGIN_PATHS` 加载绝对目录。

`package.json` 中出现包名只代表安装候选；最终以运行时 `channels?probe=true` 为准。

### 第 2 步：停止 Gateway，完成平台原生 onboarding

凭据只写入同一个隔离实例。不要把 bot token、AppSecret、二维码、OAuth credential 或 session 写进 links、浏览器本地存储、Git 或工单。

### 第 3 步：配置平台侧权限和准入

至少核对：

- bot/app 是否已启用；
- webhook、WebSocket、事件订阅、scopes/intents 是否正确；
- 精确群/频道/空间是否允许；
- 群内 sender 是否允许；
- 是否要求 `@` bot；
- 平台是否会把普通群消息实际发送给 bot。

### 第 4 步：启动 Gateway

```bash
npm start
```

### 第 5 步：运行原生 probe，并检查统一状态

Gateway 运行后，另开终端执行：

```bash
./node_modules/.bin/openclaw channels status --probe
```

随后使用第 5 节的 `cgcurl`：

```bash
cgcurl "$BASE/api/v1/health"
cgcurl "$BASE/api/v1/channels?probe=true"
cgcurl "$BASE/api/v1/delivery/status"
```

这一步必须通过，才进入主动发送和 links 配置。输出中可能包含账号或平台信息，不要原样贴入公开日志。

### 第 6 步：先做单渠道主动发送

不要急着配置 links。先调用：

```text
POST /api/v1/messages
```

确认目标真实收到消息，并记录发送成功响应中的 provider receipt/message id（若平台返回）。

### 第 7 步：从平台发一条真实入站消息

```bash
cgcurl "$BASE/api/v1/events?limit=100"
```

从 canonical event 中记录：

- `channel`
- `accountId`
- `conversationId`
- `messageId`
- 该平台实际发送使用的 `to`

`conversationId` 是入站匹配值，`to` 是出站目标；二者可能相同，但必须分别验证，不能直接假定相同。

### 第 8 步：写入 links

优先使用控制台：

```text
http://127.0.0.1:18789/channel-gateway
```

也可使用第 7 节的 `GET/PUT /api/v1/links/config`。每个 link 至少两个 endpoint。

### 第 9 步：重启后做双向验证

保存 links 只会原子写配置，不会热加载。必须重启：

```bash
# 本地
# 在运行终端 Ctrl-C，然后：
npm start

# Docker
# docker compose restart channel-gateway
```

随后从 link 内每个 endpoint 各发一条消息，检查其它 `send:true` endpoint 是否收到。

### 第 10 步：做故障恢复验证

1. 暂停一个目标账号或临时使其不可达。
2. 从源端发送消息。
3. 确认 `/api/v1/health` 或 `/api/v1/delivery/status` 出现 pending/retry。
4. 重启 Gateway。
5. 恢复目标账号。
6. 确认 pending 下降，目标实际收到消息，未出现无限回环。

完成这一步后，才开始下一个 Channel。

## 3. 推荐对接顺序

1. **Telegram**：core，外部设施最少，先验证整条 Bridge 链。
2. **Feishu**：WebSocket 默认，无需公网 webhook。
3. **Discord**：ID、权限和 intents 边界明确。
4. **Slack Socket Mode**：不需要公网 webhook。
5. **WhatsApp**：需要 QR/session，并有 message hook 特殊要求。
6. **QQ Bot**：业务常用，但要先确认 optional 安装、AppID/AppSecret、群 OpenID 和当前许可交付策略。
7. **Matrix / Mattermost / Teams / Signal**：分别依赖 homeserver、企业平台配置、Azure Bot 或 sidecar。
8. **Google Chat / LINE / SMS / Nextcloud Talk / Synology Chat**：通常需要公网 HTTPS 回调。
9. **Twitch / Tlon / Zalo / Zalo Personal / Nostr**：平台专用或能力受限，后续接入。

Raft 当前不是普通通用 links endpoint：它偏向 direct wake，最终文本不保证走当前无 Agent/LLM 的 generic outbound 路径，必须单独验证后再决定是否接入。

## 4. 首批 Channel 的具体操作

### 4.1 Telegram

1. 在 BotFather 创建 bot，取得 token。
2. Gateway 停止时，在 `openclaw.json` 配置 `channels.telegram.botToken`，或在启动/原生命令环境中注入：

```bash
export TELEGRAM_BOT_TOKEN='<BOT_TOKEN>'
```

3. Telegram 不使用 `channels login telegram`。
4. 如果要接收未 `@` bot 的普通群消息，在 BotFather 执行 `/setprivacy` → **Disable**；改变后建议把 bot 移出群再重新加入。
5. 精确配置群 chat ID。supergroup 通常是 `-100...`；群 ID 放在 `groups.<chatId>`，成员 user ID 才放在 `allowFrom/groupAllowFrom`。
6. 先做主动发送，再从群中发消息，通过 `/api/v1/events` 记录真实 `conversationId`。

本地版本文档：`node_modules/openclaw/docs/channels/telegram.md`。

### 4.2 Feishu

Gateway 停止时执行：

```bash
./node_modules/.bin/openclaw channels login --channel feishu
```

选择手工方式填写 App ID/App Secret，或使用支持的 QR 流程；随后在飞书开放平台完成 bot、事件订阅和权限。群 ID 通常为 `oc_xxx`，用户 Open ID 通常为 `ou_xxx`。

```bash
./node_modules/.bin/openclaw channels status --probe
```

本地版本文档：`node_modules/openclaw/docs/channels/feishu.md`。

### 4.3 Discord

1. 在 Discord Developer Portal 创建 Application/Bot。
2. 至少启用 **Message Content Intent**；需要角色/成员匹配时启用 Server Members Intent。
3. 邀请 bot 时授予 View Channels、Send Messages、Read Message History 等实际需要的最小权限。
4. Gateway 停止时执行：

```bash
./node_modules/.bin/openclaw channels add
```

5. 使用 Developer Mode 取得稳定的 guild/channel/user ID，不使用显示名称作为 links 目标。

本地版本文档：`node_modules/openclaw/docs/channels/discord.md`。

### 4.4 Slack

优先使用 Socket Mode，准备 App Token 与 Bot Token，再通过原生 `channels add` 写入同一实例：

```bash
./node_modules/.bin/openclaw channels add
./node_modules/.bin/openclaw channels status --probe
```

频道使用稳定 ID（通常 `C...`），用户使用稳定 ID（通常 `U...`）。确认 scopes、事件订阅和 bot 是否已加入目标频道。

本地版本文档：`node_modules/openclaw/docs/channels/slack.md`。

### 4.5 WhatsApp

建议使用独立号码。Gateway 停止时执行：

```bash
./node_modules/.bin/openclaw channels login --channel whatsapp

# 多账号示例
# ./node_modules/.bin/openclaw channels login --channel whatsapp --account work
```

QR/session 保存在隔离 credentials。远程机器应提供可靠的实时 QR 展示路径，不要依赖过期截图。launcher 在发现 WhatsApp 时要求 `channels.whatsapp.pluginHooks.messageReceived=true`；不要删除该配置。

群目标通常为 `<id>@g.us`，私聊可能为 E.164/JID；以真实入站 event 和受控 send 为准。

本地版本文档：`node_modules/openclaw/docs/channels/whatsapp.md`。

### 4.6 QQ Bot

`all` profile 当前会把 `@openclaw/qqbot` 作为 optional dependency 安装候选。开始前先确认实际存在：

```bash
test -d node_modules/@openclaw/qqbot
```

Gateway 停止时执行：

```bash
./node_modules/.bin/openclaw channels add \
  --channel qqbot \
  --token '<APP_ID>:<APP_SECRET>'
```

也可使用 `QQBOT_APP_ID`、`QQBOT_CLIENT_SECRET` 或结构化 SecretRef。群必须使用平台返回的 GROUP_OPENID，不能使用群显示名称，也不能复用另一个 bot 视角下的 OpenID。发送目标的精确格式以当前 QQ Bot 文档和受控 send 为准。

本地版本文档：`node_modules/openclaw/docs/channels/qqbot.md`。

### 4.8 微信与企业微信群

- 腾讯 `@tencent-weixin/openclaw-weixin` 当前只支持微信私聊，不能通过配置直接开启普通微信群。
- 企业微信成员群使用企业微信官方 `@wecom/wecom-openclaw-plugin@2026.5.25`，其能力声明包含 `direct` 和 `group`。本仓库已经兼容 plugin id `wecom-openclaw-plugin` 与 Channel id `wecom` 不同的情况，并在 `all` profile 自动发现该固定版本。
- 普通个人微信群只能通过桌面 RPA/hook 方案做隔离测试号 PoC，不应承诺生产稳定或使用主账号。

企业微信 Bot WebSocket 对接顺序：

1. 使用 `npm ci --omit=dev --include=optional` 安装固定版本；确认目录
   `node_modules/@wecom/wecom-openclaw-plugin` 存在。
2. 对新实例先启动一次，让 launcher 写入
   `plugins.load.paths` 和 `plugins.entries["wecom-openclaw-plugin"].enabled=true`。任何已有配置都按
   《操作手册》6.1 合并上述 load path/entry；launcher 只告警并保持旧实例可启动，不会静默改写操作员配置。
3. 停止 Gateway，在相同 `OPENCLAW_CONFIG_PATH`/隔离数据目录下执行：

   ```bash
   ./node_modules/.bin/openclaw channels add
   ```

   选择 WeCom，填写企业微信智能机器人 Bot ID 与 Secret。优先使用交互式命令，避免 Secret 进入 shell history。
4. 重新启动 Gateway，再运行 `./node_modules/.bin/openclaw channels status --probe`。只有
   `gatewayReachable: true` 的运行时 probe 才能证明账号已连接。
5. 如果平台后台已能取得 `chatid`，启动前直接配置 fail-closed 的群 allowlist；否则只允许在隔离测试租户中
   暂时使用插件默认的 `groupPolicy: "open"`，群内 `@机器人` 一次，通过
   `/api/v1/events?limit=100` 取得原始大小写不变的 `chatid`，然后立即改成：

   ```json5
   {
     channels: {
       wecom: {
         groupPolicy: "allowlist",
         groupAllowFrom: ["<WECOM_CHAT_ID>"],
         groups: {
           "<WECOM_CHAT_ID>": {
             allowFrom: ["<OPTIONAL_WECOM_USER_ID>"]
           }
         }
       }
     }
   }
   ```

   `groupAllowFrom` 放允许的群 `chatid`，不是成员 userid；只有同时设置
   `groupPolicy: "allowlist"` 才会按该列表限制群。修改后重启并复测未批准群会被拒绝。
6. 先对同一 `chatid` 调用 `/api/v1/messages` 验证主动发送，再加入 links。会话首次联系机器人之前，
   平台可能拒绝主动推送。

WeCom 与 Telegram 的最小双向 room：

```json5
{
  id: "wecom-telegram",
  endpoints: [
    {
      id: "wecom-group",
      channel: "wecom",
      accountId: "default",
      conversationId: "<WECOM_CHAT_ID>",
      to: "<WECOM_CHAT_ID>",
      receive: true,
      send: true
    },
    {
      id: "telegram-group",
      channel: "telegram",
      accountId: "default",
      conversationId: "-1001234567890",
      to: "-1001234567890",
      receive: true,
      send: true
    }
  ]
}
```

在 `plugins.entries.channel-gateway.config` 中为首个 WeCom 账号使用保守限流：

```json5
{
  deliveryAccountRateLimits: [
    { channel: "wecom", accountId: "default", ratePerSecond: 0.25, burst: 1 }
  ]
}
```

企业微信 Bot 群入站通常要求 `@机器人`；不能把它当成监听所有群普通消息。单会话回复和主动推送还受平台频率限制，扩大群范围前必须完成 429/cooldown、断线重连和重启恢复验证。
当前 rich hook patch v2 还会把 WeCom 的 `ChatType: "group"` 保留为 canonical `isGroup: true`；启动前必须通过 `npm run verify:openclaw-patch`，不能跳过 patch marker 校验。

完整判断、架构映射、限流建议和验收步骤见 [`docs/微信群接入可行性分析.md`](docs/微信群接入可行性分析.md)。

## 5. 统一 REST 调用方式

除 `/healthz` 外，业务 API 都要求 operator Bearer token。定义 helper，避免 token 进入命令行参数和 `ps`：

```bash
export BASE="http://127.0.0.1:${CHANNEL_GATEWAY_PORT:-18789}"

cgcurl() {
  curl --config - "$@" <<EOF_CURL
header = "Authorization: Bearer ${CHANNEL_GATEWAY_TOKEN}"
EOF_CURL
}
```

常用读取：

```bash
curl -fsS "$BASE/healthz"
cgcurl "$BASE/api/v1/health"
cgcurl "$BASE/api/v1/channels?probe=true"
cgcurl "$BASE/api/v1/delivery/status"
cgcurl "$BASE/api/v1/links"
cgcurl "$BASE/api/v1/events?limit=100"
```

成功响应统一为：

```json
{ "ok": true, "result": {} }
```

Bridge 失败响应统一为：

```json
{
  "ok": false,
  "error": {
    "code": "ERROR_CODE",
    "retryable": true,
    "retryAfterMs": 1000
  }
}
```

注意：认证失败由 OpenClaw Gateway auth 层返回，不一定使用上述 Bridge envelope。

## 6. 主动发送消息

### 6.1 文本

```bash
cgcurl -X POST -H 'Content-Type: application/json' \
  -d '{
    "channel":"telegram",
    "accountId":"default",
    "to":"-1001234567890",
    "message":"Channel Gateway outbound smoke",
    "idempotencyKey":"telegram-smoke-20260713-001"
  }' \
  "$BASE/api/v1/messages"
```

必填字段只有：

- `channel`
- `to`
- `idempotencyKey`

同一次逻辑发送在网络超时或结果未知后重试时，必须复用原 `idempotencyKey`；新的逻辑消息必须生成新 key。

### 6.2 媒体和回复

```bash
cgcurl -X POST -H 'Content-Type: application/json' \
  -d '{
    "channel":"feishu",
    "accountId":"default",
    "to":"oc_xxx",
    "message":"附件测试",
    "mediaUrls":["https://example.invalid/controlled-test.png"],
    "replyToId":"<PROVIDER_MESSAGE_ID>",
    "threadId":"<OPTIONAL_THREAD_ID>",
    "idempotencyKey":"feishu-media-20260713-001"
  }' \
  "$BASE/api/v1/messages"
```

是否支持 `mediaUrls`、reply、thread、voice、silent 等字段由目标 Channel 决定。Gateway 只做字段白名单和类型校验；真实平台能力必须逐项 live 验证。

外部应用不要直连内部 `dispatchGatewayMethod`。稳定公开入口是 `/api/v1`；内部映射才是：

```text
REST /api/v1/messages → Gateway RPC send → Channel outbound adapter
REST /api/v1/channels → Gateway RPC channels.status/start/stop/logout
```

## 7. 读取和更新 links

### 7.1 推荐：网页控制台

访问：

```text
http://127.0.0.1:18789/channel-gateway
```

Token 只保存在页面内存。连接后读取 Channel 状态、添加互通房间、保存 links；保存成功后必须重启服务。

### 7.2 REST：先 GET revision，再 PUT

先读取当前完整配置：

```bash
cgcurl "$BASE/api/v1/links/config" > /tmp/channel-gateway-links-current.json
```

用 Node 22 读取当前 revision 并生成更新请求。按真实 ID 修改 endpoints：

```bash
node --input-type=module <<'NODE'
import { readFileSync, writeFileSync } from 'node:fs';

const envelope = JSON.parse(
  readFileSync('/tmp/channel-gateway-links-current.json', 'utf8'),
);
if (envelope.ok !== true) {
  throw new Error(`GET links/config failed: ${JSON.stringify(envelope.error)}`);
}

const request = {
  revision: envelope.result.revision,
  // 删除此字段表示保持当前值；null 表示清除持久化 override。
  deliveryMaxConcurrency: envelope.result.deliveryMaxConcurrency,
  links: [
    {
      id: 'room-main',
      endpoints: [
        {
          id: 'telegram-main',
          channel: 'telegram',
          accountId: 'default',
          conversationId: '-1001234567890',
          to: '-1001234567890',
          receive: true,
          send: true,
        },
        {
          id: 'feishu-main',
          channel: 'feishu',
          accountId: 'default',
          conversationId: 'oc_replace_me',
          to: 'oc_replace_me',
          receive: true,
          send: true,
        },
      ],
    },
  ],
};

writeFileSync(
  '/tmp/channel-gateway-links-update.json',
  `${JSON.stringify(request, null, 2)}\n`,
  { mode: 0o600 },
);
NODE
```

提交：

```bash
cgcurl -X PUT -H 'Content-Type: application/json' \
  --data-binary @/tmp/channel-gateway-links-update.json \
  "$BASE/api/v1/links/config"
```

规则：

- 顶层只允许 `links`、`revision`、可选 `deliveryMaxConcurrency`。
- 成功响应的 `result.restartRequired` 为 `true`，必须重启。
- 两个管理员并发编辑时，旧 revision 返回 HTTP `409`、`CONFIG_CONFLICT`；重新 GET 后再合并，不能盲目覆盖。
- API/控制台可以在运行时原子写配置，但不会热加载。
- 直接编辑 `openclaw.json` 前应停止 Gateway；不要同时混用直接编辑、控制台和 API。

### 7.3 links 不变量

- link `id` 全局唯一。
- 一个 link 至少两个 endpoint。
- 同一 link 内 endpoint `id` 唯一。
- `channel + accountId + conversationId` 不能重复。
- 同一 `channel + conversationId` 不能由不同 bot account 同时拥有。
- `channel + accountId + to` 不能重复。
- `receive`/`send` 默认均为 `true`；关闭其中之一可形成单向边界。
- 一个 bot 可以负责多个群；同一个群只能由一个 bot account 拥有。

## 8. 入站事件、SSE 和 ACK

### 8.1 REST 拉取

```bash
cgcurl "$BASE/api/v1/events?limit=100"
```

`limit` 范围为 1–500。`after=<eventId>` 只能使用数据库中已知的 cursor。

### 8.2 SSE

```bash
cgcurl -N "$BASE/api/v1/events/stream"
```

事件帧：

```text
event: message
id: evt_xxx
data: {"id":"evt_xxx","channel":"feishu",...}

: heartbeat
```

消费顺序必须是：

```text
读取 SSE event
→ 按 event.id 幂等处理
→ 写入自己的 durable store 并提交
→ POST /api/v1/events/:eventId/ack
```

ACK：

```bash
cgcurl -X POST -H 'Content-Type: application/json' -d '{}' \
  "$BASE/api/v1/events/evt_xxx/ack"
```

注意：

- 连接会先 replay 所有仍 pending 的事件，再接收实时事件。
- 当前 stream 不使用 `Last-Event-ID` 做过滤；客户端仍要按 event id 去重。
- 慢消费者队列溢出会断开；重连后依赖 pending replay 恢复。
- 浏览器原生 `EventSource` 不能自定义 Authorization header，不应直接用于这个 operator API；使用可设置 header 的 Node `fetch`/SSE 客户端，或由受信任同源后端代理。
- 只在下游 durable commit 成功后 ACK；不要“先 ACK 再处理”。

## 10. 常见故障定位

| 现象 | 首要检查 |
| --- | --- |
| 启动直接拒绝 | `npm run verify:openclaw-patch`、Node/OpenClaw 固定版本、已有配置的 load path/entry |
| `/healthz` 不通 | 进程、端口、bind、防火墙、容器日志 |
| `/api/v1/*` 返回 401 | 当前 shell token 与启动实例 token 是否一致 |
| `/api/v1/health` 返回 503 | 数据目录/SQLite 权限、磁盘、运行时 degraded code |
| Channel 未出现在 probe | 包是否实际安装、插件是否启用、凭据是否写入同一实例 |
| 主动发送失败 | `channel/accountId/to`、平台账号状态、provider 限流和返回 error code |
| 收不到入站 | 平台事件订阅、精确群准入、sender allowlist、mention/privacy、webhook/sidecar |
| 入站可见但不转发 | endpoint 的 `channel/accountId/conversationId` 是否精确匹配，是否已重启 |
| 只发到部分目标 | `/api/v1/delivery/status`、目标账号 probe、pending/retry/cooldown |
| 保存 links 返回 409 | revision 已过期，重新 GET 后合并再 PUT |
| 重复回环 | 检查是否绕过 Gateway sender、是否人为改写 marker、provider 是否缺少稳定 receipt；保留现场再排查 |

## 11. 当前证据边界

仓库测试能证明统一 API、事件持久化、links fan-out、worker retry/lease、SSE/ACK、echo suppression 和 fake-provider 恢复逻辑；不能替代真实平台的 credential、webhook、网络、权限、群 ID、媒体、reply、receipt 和限流验证。

开始逐一对接时，建议先复制第 9 节验收模板，并严格一次只增加一个 Channel 或一个 account。
