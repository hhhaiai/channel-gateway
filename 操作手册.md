# Channel Gateway 操作手册

> 面向首次部署者和运维人员。本手册以当前仓库的 launcher、Bridge API、Docker Compose 和固定 Host 为准；运行时版本必须是 `openclaw@2026.6.11`。

## 1. 适用范围与边界

Channel Gateway 在 OpenClaw 的 `before_dispatch` 阶段接收已通过 Channel 准入检查的入站消息，先持久化到 SQLite，再按 `links` 分发到其他会话。入站消息被标记为 handled，**不会进入 Agent/LLM**。

- 可作为 Telegram、Discord、Feishu、Slack、WhatsApp 之间的群聊或私聊互通服务；QQ、Teams、Matrix、Signal 需额外、显式安装插件或 sidecar。
- 它不实现或替代各平台登录、协议、媒体、限流和平台权限；这些由固定版本的 OpenClaw 和 provider 插件处理。
- `links` 不是租户隔离或授权边界。不同不可信租户必须拆分为不同实例、数据目录、token、端口、系统用户/容器与网络策略。
- 没有真实平台凭据、平台侧事件订阅/权限和必要 sidecar 时，只能验证本地服务及 contract/fake 测试，不能宣称真实 provider 已可收发。

## 2. 前置条件与安全要求

### 2.1 软件与网络

- Node.js：`>=22.22.2 <23`。本机可用 `node --version` 确认。
- npm、Git；Docker 部署还需要 Docker Engine 与 Docker Compose v2。
- 每个 provider 的真实凭据、网络访问以及平台后台所需的事件订阅/权限；Signal 还需要已注册的 `signal-cli` sidecar。
- 生产环境仅将服务暴露给受信任网络。Compose 默认监听 LAN，必须在前方加 TLS 反向代理、防火墙与来源 ACL。

### 2.2 凭据和 token

operator Bearer token 可以发送消息、启动/停止/登出 Channel，应按管理员凭据保护。不要把 token、二维码 session、provider credential 或数据目录提交到 Git、镜像、Compose 文件或日志。

以下示例把 token 放在环境变量，并由 `curl --config -` 经 stdin 传递 header，避免 token 出现在命令行参数或进程列表中：

```bash
export CHANNEL_GATEWAY_TOKEN="$(openssl rand -hex 32)"
# 不要执行：curl -H "Authorization: Bearer $CHANNEL_GATEWAY_TOKEN" ...
```

自动化部署应从 secret manager 注入。launcher 未收到 `CHANNEL_GATEWAY_TOKEN` 时会生成权限为 `0600` 的 token 文件；生产部署建议显式注入并妥善保存。

## 3. 数据目录、运行 profile 与端口

每个实例使用独立数据目录，首次启动由 launcher 创建：

```text
$CHANNEL_GATEWAY_DATA_DIR/
├── config/openclaw.json
├── state/channel-gateway.sqlite
├── credentials/gateway-token
└── workspace/
```

profile：

| profile | 内容 |
| --- | --- |
| `base` | OpenClaw core 与内置 Telegram 等能力；不安装本仓库 optional Channel 包。 |
| `common` | 另安装精确版本 `2026.6.11` 的 Discord、Feishu、Slack、WhatsApp 插件。 |

固定 Host、所有官方 Channel 插件与 rich `before_dispatch` patch 是一个兼容单元。启动前 launcher 会检查 Host 版本和 patch marker；不匹配会拒绝启动。

常用环境变量：

| 变量 | 默认/说明 |
| --- | --- |
| `CHANNEL_GATEWAY_DATA_DIR` | 实例根目录；应为本实例私有、可写的绝对路径。 |
| `CHANNEL_GATEWAY_TOKEN` | operator Bearer token。 |
| `CHANNEL_GATEWAY_BIND` | `loopback`、`lan`、`tailnet`、`auto` 或 `custom`；本地推荐 `loopback`。 |
| `CHANNEL_GATEWAY_PORT` | 1–65535，默认 `18789`。 |
| `CHANNEL_GATEWAY_CUSTOM_BIND_HOST` | 仅 `custom` bind 必填。 |
| `CHANNEL_GATEWAY_PLUGIN_PATHS` | 外置插件的**绝对路径**，多路径用操作系统 path delimiter（Linux/macOS 为 `:`）分隔。 |

## 4. 本地首次启动（base 或 common）

在仓库 `channel-gateway/` 中执行。先选择一个 profile，不能连续执行两条 `npm ci`：

```bash
cd /Users/sanbo/Desktop/openclaw/channel-gateway

# base：仅 core/Telegram 等
npm ci --omit=dev --omit=optional

# common：改用此条以安装 Discord、Feishu、Slack、WhatsApp
# npm ci --omit=dev --include=optional

npm run verify:openclaw-patch
export CHANNEL_GATEWAY_DATA_DIR="$PWD/.channel-gateway"
export CHANNEL_GATEWAY_TOKEN="$(openssl rand -hex 32)"
export CHANNEL_GATEWAY_BIND=loopback
npm start
```

另开终端做最小就绪检查：

```bash
curl http://127.0.0.1:18789/healthz
```

首次启动确认成功后，在运行终端按 `Ctrl-C` 正常停止，再进行 provider onboarding。不要手工替换 launcher 生成的数据库、workspace 或 Bridge 配置；已有 `config/openclaw.json` 不会被 launcher 静默覆盖，并会被校验其隔离、插件、端口与 Control UI/reload 约束。

## 5. Docker Compose 部署

### 5.1 构建与启动

在仓库目录中执行。token 由环境变量提供，绝不写进 compose 文件：

```bash
cd /Users/sanbo/Desktop/openclaw/channel-gateway
export CHANNEL_GATEWAY_TOKEN="$(openssl rand -hex 32)"

# base
CHANNEL_PROFILE=base docker compose build
env CHANNEL_PROFILE=base CHANNEL_GATEWAY_TOKEN="$CHANNEL_GATEWAY_TOKEN" docker compose up -d

# common（如需官方常用插件，改用此组）
# CHANNEL_PROFILE=common docker compose build
# env CHANNEL_PROFILE=common CHANNEL_GATEWAY_TOKEN="$CHANNEL_GATEWAY_TOKEN" docker compose up -d
```

Compose 使用非 root `node` 用户、只读根文件系统、`/data` 下四个持久 named volume 与受限 tmpfs；对外映射端口默认为 `18789`。查看状态：

```bash
docker compose ps
docker compose logs --tail=200 channel-gateway
curl http://127.0.0.1:18789/healthz
```

`/healthz` 只表示 OpenClaw HTTP 服务可达；还必须用带 token 的 `/api/v1/health` 验证 Bridge/SQLite readiness（见第 9 节）。

### 5.2 容器中的首次配置

首次运行生成持久 volumes 后，必须先停止 Gateway，再用共享 named volumes 和 Compose 环境启动一个**不启动 Gateway**的一次性容器执行 onboarding。不能在运行中的 Gateway 上使用 `docker compose exec`，也不能让两个进程并发写同一个 `openclaw.json` 或 credentials 目录。不要向 shell 显示或复制 token；Compose 会从当前环境将它注入一次性容器。

```bash
docker compose stop channel-gateway
docker compose run --rm --no-deps channel-gateway sh -lc '
  ./node_modules/.bin/openclaw channels add
  ./node_modules/.bin/openclaw channels status --probe
'
```

不同 provider 的登录命令见第 6 节。配置完成后：

```bash
docker compose up -d channel-gateway
docker compose logs --tail=200 -f channel-gateway
```

## 6. 首次 Channel onboarding

本地部署时，停止 Gateway 后，导出同一实例环境，再让 OpenClaw 原生命令写入该实例的隔离配置：

```bash
export OPENCLAW_HOME="$CHANNEL_GATEWAY_DATA_DIR"
export OPENCLAW_CONFIG_PATH="$CHANNEL_GATEWAY_DATA_DIR/config/openclaw.json"
export OPENCLAW_STATE_DIR="$CHANNEL_GATEWAY_DATA_DIR/state"
export OPENCLAW_WORKSPACE_DIR="$CHANNEL_GATEWAY_DATA_DIR/workspace"
export OPENCLAW_OAUTH_DIR="$CHANNEL_GATEWAY_DATA_DIR/credentials"
export OPENCLAW_GATEWAY_TOKEN="$CHANNEL_GATEWAY_TOKEN"
export OPENCLAW_GATEWAY_PORT="${CHANNEL_GATEWAY_PORT:-18789}"

./node_modules/.bin/openclaw channels add
./node_modules/.bin/openclaw channels status --probe
```

按平台补充：

- **Telegram**：core 自带；在原生配置/环境中配置 BotFather token，不使用 `channels login telegram`。
- **Discord**：`common` profile；通过 `channels add` 配置 bot token、guild/channel allowlist 与 intents。
- **Feishu**：`common` profile；`./node_modules/.bin/openclaw channels login --channel feishu`，并在开放平台配置 app credential、事件订阅和权限。
- **Slack**：`common` profile；通过 `channels add` 配置 app/bot token、socket/events 与 scopes。
- **WhatsApp**：`common` profile；`./node_modules/.bin/openclaw channels login --channel whatsapp` 完成 QR/session；必要时使用独立 account id。
- **Teams/Matrix/Signal**：按固定 OpenClaw 版本的原生文档配置；外置插件必须经 `CHANNEL_GATEWAY_PLUGIN_PATHS` 指向绝对路径。Signal 要先准备 sidecar，并将其持久目录单独备份。

完成后使用 `channels status --probe`，但不要把其包含的凭据内容贴到工单或日志。

### 6.1 QQ 显式安装

QQ 不随 base/common 镜像发布，通常还需要人工许可审查与额外 sidecar。审批后才在服务树外固定位置安装。目录创建可由管理员完成，依赖安装必须由非 root 的专用服务用户执行：

```bash
sudo install -d -m 0750 -o channel-gateway -g channel-gateway /opt/channel-plugins/qq
# 切换到 channel-gateway 服务用户后：
cd /opt/channel-plugins/qq
npm init -y
npm pkg set private=true --json
npm pkg set 'dependencies.@openclaw/qqbot=2026.6.11'
npm install --package-lock-only --ignore-scripts
# 审查 package-lock 的来源、integrity、许可后才执行：
npm ci --omit=dev
export CHANNEL_GATEWAY_PLUGIN_PATHS=/opt/channel-plugins/qq/node_modules/@openclaw/qqbot
```

Docker 应构建含该插件的派生镜像，不要在运行中的只读基础镜像临时安装。

## 7. 群消息准入与普通群消息

Bridge 只能接收已通过 OpenClaw 的 access/mention gate 的消息。要转发未 `@` bot 的普通群消息，需要同时设置精确群、群内发件人准入与平台侧隐私策略：

1. 从平台后台/API、群信息页或受控日志取得真实 `conversationId`；先让成员 `@` bot 一次可帮助发现 ID。QQ 用群 OpenID，不能用显示名称；Telegram supergroup 使用负数 chat ID。
2. 在 `config/openclaw.json` 的 `channels.<provider>.groups.<conversationId>` 配置该精确群，并设置 `requireMention: false`。
3. 用 `groupPolicy`、全局 `groupAllowFrom` 或 `groups.<id>.allowFrom` 配置**群内成员 ID/openid**。它们不是群 ID 列表。
4. Telegram 在 BotFather 中执行 `/setprivacy`，选择 **Disable**；否则 Telegram 不会把未 @ 的普通群消息交给 bot。
5. 默认收紧群与 sender allowlist。只有确认风险后，且 provider schema 支持时，才对已批准的精确群设 `groupPolicy: "open"`。

已核验的 Telegram/QQ 形状如下；Feishu、WhatsApp 等字段以固定版本的原生 schema 为准：

```json5
{
  channels: {
    telegram: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["<TELEGRAM_USER_ID>"],
      groups: {
        "-1001234567890": {
          requireMention: false,
          allowFrom: ["<TELEGRAM_USER_ID>"]
        }
      }
    },
    qqbot: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["<QQ_MEMBER_OPENID>"],
      groups: {
        "<QQ_GROUP_OPENID>": { requireMention: false }
      }
    }
  }
}
```

私聊同样应设置明确的 DM/pairing policy；不要把所有群或所有发件人直接放开以解决“收不到消息”。

## 8. 配置 links 并验证双向分发

在 `config/openclaw.json` 的 `plugins.entries.channel-gateway.config.links` 配置。每个 endpoint 的 `conversationId` 用于匹配入站事件，`to` 用于 OpenClaw `send` RPC；两者可能相同，但必须分别通过真实 Channel status/event 核验。

```json5
{
  plugins: {
    entries: {
      "channel-gateway": {
        enabled: true,
        hooks: { allowConversationAccess: true, timeouts: { before_dispatch: 5000 } },
        config: {
          // 保留 launcher 首次生成的 databasePath，不要把 Docker 的 /data 路径复制到本地实例。
          links: [{
            id: "support-global",
            endpoints: [
              { id: "qq", channel: "qqbot", accountId: "default", conversationId: "<QQ_GROUP_OPENID>", to: "<QQ_GROUP_OPENID>", receive: true, send: true },
              { id: "feishu", channel: "feishu", accountId: "default", conversationId: "<FEISHU_CONVERSATION_ID>", to: "<FEISHU_SEND_TARGET>", receive: true, send: true },
              { id: "whatsapp", channel: "whatsapp", accountId: "default", conversationId: "<WHATSAPP_CHAT_ID>", to: "<WHATSAPP_SEND_TARGET>", receive: true, send: true },
              { id: "telegram", channel: "telegram", accountId: "default", conversationId: "-1001234567890", to: "-1001234567890", receive: true, send: true }
            ]
          }]
        }
      }
    }
  }
}
```

- `receive` 和 `send` 默认为 `true`；将任一端设为 `false` 可构成单向边界。
- 编辑配置后重启服务。入站 event 与 fan-out delivery 会在同一 SQLite transaction 创建，重启后继续投递。
- marker、delivery receipt 与 scoped message identity 用于 echo suppression；不要手动伪造 marker。
- 对一个已批准群做受控发送并观察所有目标群、events 与 receipt；再令一个目标暂时不可达，确认 pending delivery 在重启并恢复目标后下降，才算验证了重启恢复。

## 9. REST、SSE 与 ACK

除 `/healthz` 外，以下接口均需要 operator Bearer token。先定义安全的 curl helper：

```bash
BASE=http://127.0.0.1:18789
cgcurl() {
  curl --config - "$@" <<EOF_CURL
header = "Authorization: Bearer ${CHANNEL_GATEWAY_TOKEN}"
EOF_CURL
}

curl "$BASE/healthz"
cgcurl "$BASE/api/v1/health"
cgcurl "$BASE/api/v1/channels?probe=true"
cgcurl "$BASE/api/v1/links"
cgcurl "$BASE/api/v1/events?limit=100"
```

常用控制命令：

```bash
# 受控发送；每次逻辑发送生成唯一 idempotencyKey。若请求结果未知而重试同一逻辑发送，必须复用原 key。
cgcurl -X POST -H 'Content-Type: application/json' \
  -d '{"channel":"telegram","accountId":"default","to":"-1001234567890","message":"hello","mediaUrls":[],"idempotencyKey":"operator-20260711-1"}' \
  "$BASE/api/v1/messages"

cgcurl -X POST -H 'Content-Type: application/json' -d '{"accountId":"default"}' "$BASE/api/v1/channels/telegram/start"
cgcurl -X POST -H 'Content-Type: application/json' -d '{"accountId":"default"}' "$BASE/api/v1/channels/telegram/stop"
cgcurl -X POST -H 'Content-Type: application/json' -d '{"accountId":"default"}' "$BASE/api/v1/channels/telegram/logout"

# 在消费者完成自己的 durable commit 后再 ACK；ACK 幂等。
cgcurl -X POST -H 'Content-Type: application/json' -d '{}' "$BASE/api/v1/events/evt_123/ack"

# SSE：重放所有尚 pending 的事件，再接收实时事件与 heartbeat。
cgcurl -N -H 'Last-Event-ID: evt_122' "$BASE/api/v1/events/stream"
```

`Last-Event-ID` 只是观察水位，不能跳过较早 pending 事件。慢消费者会被有界队列断开；客户端应重新连接并依赖 pending replay 恢复。SQLite commit 失败时，Bridge 仍会阻止消息进入 LLM，认证 `/api/v1/health` 会返回 `503`；应修复磁盘/权限再重启，不能绕过 durable store。

## 10. 日常检查、备份与恢复

### 10.1 日常检查

```bash
npm run verify:openclaw-patch
./node_modules/.bin/openclaw channels status --probe
# 运行中再执行第 9 节 health/channels/links/events 请求
```

本地正常停止使用 `Ctrl-C`；Docker 使用：

```bash
docker compose stop channel-gateway
docker compose start channel-gateway
```

### 10.2 备份

备份四个数据子目录：`config/`、`state/`、`credentials/`、`workspace/`。最佳方式是停止服务后对整个数据目录做一致快照：

```bash
# 本地示例：先停止 npm start，再执行；目标目录必须由受信任备份系统保护。
tar -C "$(dirname "$CHANNEL_GATEWAY_DATA_DIR")" \
  -czf "channel-gateway-backup-$(date +%F).tgz" \
  "$(basename "$CHANNEL_GATEWAY_DATA_DIR")"
```

热备份 SQLite 时必须使用 SQLite backup API，或同时一致地处理 `channel-gateway.sqlite`、`-wal`、`-shm`；**不能只复制主数据库文件**。Docker 部署应对四个 named volume 做一致快照，并将 credentials 作为高敏感备份加密保管。

### 10.3 恢复

1. 停止实例，保留当前数据目录副本以便回滚。
2. 恢复匹配版本的四个目录/volume，确保不会混用不同租户的数据。
3. 检查敏感权限：数据目录为 `0700`，token 文件为 `0600`。
4. 运行 `npm run verify:openclaw-patch`，启动后检查 `/healthz`、认证 health、`channels?probe=true`，并做一次受控重启恢复验证。

## 11. 升级、许可证与 SBOM

Host 和 Channel plugin 必须整体升级；禁止只升级其中一项：

1. 在变更分支修改所有精确版本与 `OPENCLAW_VERSION`，重建 `package-lock.json`。
2. 按新上游 source/dist 重新生成 `patches/openclaw-v*.patch`，不能把旧 patch 强套到新 bundle。
3. 执行 `npm ci && npm run patch:openclaw && npm run verify:openclaw-patch`。
4. 执行 `npm test`、`npm run check`、许可证/SBOM、base/common image build 和 standalone restart smoke。
5. 用真实 provider credential 做 channels probe 与受控收发；完成后才滚动生产。保留升级前备份以便回滚。

许可证与交付证据：

```bash
npm run licenses          # artifacts/licenses.json；普通模式只报告 blocker
npm run licenses:strict   # UNLICENSED/缺 license metadata 时失败
npm run sbom              # artifacts/sbom.cdx.json，CycloneDX
```

`artifacts/` 被 Git 忽略，应作为 CI 构建 artifact 保存。GPL/LGPL/MPL 需要人工合规审查；官方 optional Channel 的 metadata 可能缺 license，strict 的保守失败需要结合 `licenses/openclaw/LICENSE`、`licenses/openclaw/THIRD_PARTY_NOTICES.md` 和供应方证据处理。`npm run sbom` 首次需要访问 registry 获取固定的 `@cyclonedx/cyclonedx-npm@6.0.0`；不要手工编辑或伪造 SBOM。

## 12. 故障排查

| 现象 | 检查顺序与处理 |
| --- | --- |
| 启动拒绝或 patch 错误 | 执行 `npm run verify:openclaw-patch`；确认 Host 精确为 `openclaw@2026.6.11`，不要跳过 patch 检查。 |
| `/healthz` 不通 | 查看启动输出或 `docker compose logs --tail=200 channel-gateway`；确认端口、bind、防火墙及进程/容器状态。 |
| 认证 API 返回 401 | 确认当前 shell 的 `CHANNEL_GATEWAY_TOKEN` 与该实例启动 token 一致；不要把 token 放入 URL 或日志。 |
| `/api/v1/health` 为 503 | 检查数据目录、SQLite 文件、剩余磁盘、目录权限；修复后重启。此状态 fail-closed，不能跳过 SQLite。 |
| 收不到群普通消息 | 检查精确 `groups.<conversationId>`、`requireMention:false`、sender admission；Telegram 还需 BotFather `/setprivacy` Disable。确认群 ID 不是显示名称或成员 ID。 |
| 只发到部分目标/有 pending delivery | 查 `/api/v1/health` 的 deliveries、`/api/v1/events`，再跑 `channels?probe=true`；修复目标 provider 后重启并观察 pending 是否恢复。 |
| 外置插件未加载 | 确认 `CHANNEL_GATEWAY_PLUGIN_PATHS` 是绝对路径、路径分隔正确，且插件 manifest 的 Host/API 兼容、Channel id 无冲突。 |
| Docker 健康但互通失败 | `/healthz` 只检查 HTTP；使用认证 `/api/v1/health`、`/channels?probe=true`、`/links` 检查 Bridge、SQLite、provider 与 links。 |

## 13. 上线验收清单

- [ ] Node 或镜像版本符合要求，`npm run verify:openclaw-patch` 通过。
- [ ] 每个不可信租户独立实例、token、数据目录与网络策略；token 未进入 Git/镜像/日志。
- [ ] `/healthz` 与认证 `/api/v1/health` 均正常，`channels?probe=true` 通过。
- [ ] 各 provider 在其平台后台完成真实凭据、权限、事件订阅（及必要 sidecar）配置。
- [ ] 群 ID、sender admission、`requireMention:false` 已按 provider 核验；Telegram privacy 已 Disable。
- [ ] 每个 link 的 `conversationId` 与 `to` 都通过真实事件/status 验证；双向、echo suppression、reply relation 按业务场景受控验收。
- [ ] 已测试目标暂时不可达后的 pending delivery、重启和恢复；已完成加密备份与恢复演练。
- [ ] 已保存 `licenses`、`licenses:strict`（如适用）与 `sbom` 的 CI artifact，并完成必要人工合规审查。
